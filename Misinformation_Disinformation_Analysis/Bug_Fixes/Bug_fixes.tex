\textbf{\faInfoCircle\quad Abstract.}\quad
This report documents the results of a comprehensive pre-publication technical
audit of the \emph{MisDetect} OWL~2~DL ontology
(\texttt{misinformation\_ontology.ttl}).
A total of \textbf{13 issues} were identified across the categories of
\emph{structural correctness}, \emph{authorship \& metadata},
\emph{individual completeness}, and \emph{semantic modelling}.
All issues were corrected in-place; the corrected file was validated
using the Raptor RDF parser (zero parse errors, 769 triples).
Triple count increased from 732 to 769, reflecting the additions introduced
by the bug fixes and publication-readiness enhancements.
\end{infobox}

\section{Overview}
% ============================================================

The \emph{MisDetect} ontology provides a formal OWL~2~DL schema for
representing, classifying, and reasoning about misinformation in
Twitter/X data across three thematic domains: Political, Educational,
and Healthcare. Prior to submission for publication, a full technical
audit was conducted covering:

\begin{itemize}[leftmargin=1.4em, itemsep=2pt]
  \item Turtle serialisation syntax validity;
  \item OWL~2~DL axiom correctness and consistency with declared functional properties;
  \item Completeness of illustrative individuals against their class domain constraints;
  \item Accuracy of derived numeric metadata;
  \item Conformance of authorship metadata to publication standards
        (Dublin Core, DCTerms, FOAF, PROV-O); and
  \item Semantic annotation quality (\texttt{rdfs:comment}, \texttt{skos:scopeNote},
        \texttt{skos:definition}).
\end{itemize}

\subsection{Severity Classification}

Issues are classified using the following four-level scheme:

\vspace{4pt}
\begin{tabular}{@{} l L{11cm} @{}}
\toprule
\textbf{Severity} & \textbf{Definition} \\
\midrule
\badgeCritical & Causes OWL~2 inconsistency, violates a declared functional property,
                 or would be rejected by a reasoner or conformant parser. \\[4pt]
\badgeMajor    & Incorrect computed data, incomplete individuals, or metadata
                 that misattributes the work. \\[4pt]
\badgeMinor    & Suboptimal serialisation or annotation practice that does not
                 affect reasoning but reduces tool compatibility. \\[4pt]
\badgeEnhancement & Publication-readiness additions that are not errors but are
                    required for submission to a Semantic Web venue. \\
\bottomrule
\end{tabular}

% ── Bug 1 ────────────────────────────────────────────────────
\subsection{Bug~\#1 --- Missing \texttt{hasAuthor} on \texttt{tweet\_002}}
\label{bug:1}

\textbf{Severity:} \badgeCritical\quad
\textbf{Category:} Individual Completeness\quad
\textbf{Status:} Fixed

\textbf{Description.}
The individual \ind{tweet\_002} (declared as \texttt{mis:Retweet}) had no
\prop{hasAuthor} triple.
\prop{hasAuthor} is declared both as an \texttt{owl:FunctionalProperty} with
\texttt{rdfs:domain mis:Tweet}, meaning every \texttt{mis:Tweet} individual must
carry exactly one value.
A reasoner (e.g., HermiT, Pellet) would classify the knowledge base as
\emph{inconsistent} under closed-world assumptions if this constraint is treated
as a cardinality restriction, and tools such as Protégé would flag the individual.

\textbf{Root cause.}
\ind{tweet\_002} was added as a minimal example retweet without mirroring the
full property pattern established for \ind{tweet\_001}.

\textbf{Fix applied.}
A new supporting individual \ind{user\_retweeter\_01} (class \texttt{mis:UnverifiedUser})
was added to the individuals section, and \ind{tweet\_002} was updated:

\begin{lstlisting}
misind:tweet_002
    ...
    mis:hasAuthor misind:user_retweeter_01 ;
    ...
\end{lstlisting}

% ── Bug 3 ────────────────────────────────────────────────────
\subsection{Bug~\#3 --- Missing PROV-O Timestamp on \texttt{tweet\_002}}
\label{bug:3}

\textbf{Severity:} \badgeMajor\quad
\textbf{Category:} Individual Completeness\quad
\textbf{Status:} Fixed

\textbf{Description.}
\ind{tweet\_001} carried \texttt{prov:generatedAtTime} to record when it was
published. \ind{tweet\_002}, despite being a \texttt{prov:Entity} subclass
(\texttt{mis:Tweet} $\sqsubseteq$ \texttt{prov:Entity}), omitted this triple,
breaking the provenance pattern and reducing PROV-O interoperability.

\textbf{Fix applied.}
\begin{lstlisting}
misind:tweet_002
    ...
    prov:generatedAtTime "2024-06-15T12:45:00"^^xsd:dateTime .
\end{lstlisting}

% ── Bug 4 ────────────────────────────────────────────────────
\subsection{Bug~\#4 --- Incorrect \texttt{dc:creator} Value}
\label{bug:4}

\textbf{Severity:} \badgeMajor\quad
\textbf{Category:} Authorship Metadata\quad
\textbf{Status:} Fixed

\textbf{Description.}
The ontology header contained:
\begin{lstlisting}
dc:creator "CAIR Nepal"^^xsd:string ;
\end{lstlisting}
Dublin Core \texttt{dc:creator} is intended to identify the \emph{individual}
responsible for creating the intellectual content, not the organisational
affiliation.
Assigning an organisation name here misattributes the work in bibliographic
databases and violates best-practice ontology metadata guidelines
(e.g., the Linked Open Vocabularies (LOV) submission checklist).

\textbf{Fix applied.}
\begin{lstlisting}
dc:creator        "Samriddha Pathak"^^xsd:string ;
dc:contributor    "Samriddha Pathak (CAIR Nepal Research Group)"^^xsd:string ;
dcterms:creator   misind:SamriddhaPathak ;
\end{lstlisting}
The publisher entry \texttt{dc:publisher "CAIR Nepal"} was retained unchanged,
as it correctly identifies the publishing institution.

% ── Bug 7 ────────────────────────────────────────────────────
\subsection{Bug~\#7 --- Duplicate \texttt{rdfs:comment} on Ontology IRI}
\label{bug:7}

\textbf{Severity:} \badgeMinor\quad
\textbf{Category:} Serialisation Quality\quad
\textbf{Status:} Fixed

\textbf{Description.}
The ontology IRI carried two distinct \texttt{rdfs:comment} triples.
While Turtle and RDF allow multiple values for non-functional properties,
many ontology browsers (e.g., Protégé's Annotation view, WebVOWL) display
only one comment and silently suppress the second, causing the vocabulary
reuse note to be invisible to users.
Furthermore, semantically, the second comment described \emph{vocabulary
alignment} (a scope note), not a human-readable description, making
\texttt{skos:scopeNote} the more precise property.
\newpage
\textbf{Fix applied.}
The second \texttt{rdfs:comment} was changed to \texttt{skos:scopeNote}:
\begin{lstlisting}
skos:scopeNote
    "This ontology reuses concepts from SIOC (sioc:Post,
     sioc:UserAccount), PROV-O (prov:Entity, prov:Activity,
     prov:Agent), NIF (nif:String, nif:Context), OA
     (oa:Annotation, oa:TextualBody), and Schema.org
     (schema:Person, schema:Place)."^^xsd:string .
\end{lstlisting}

% ── Bug 8 ────────────────────────────────────────────────────
\subsection{Bug~\#8 --- \texttt{mis:Thread} Missing \texttt{rdfs:subClassOf}}
\label{bug:8}

\textbf{Severity:} \badgeMinor\quad
\textbf{Category:} Class Model Consistency\quad
\textbf{Status:} Fixed

\textbf{Description.}
Every entity-level class in Part~1 of the ontology
(\texttt{mis:Tweet}, \texttt{mis:User}, \texttt{mis:PropagationCascade},
\texttt{mis:Dataset}, etc.) carries \texttt{rdfs:subClassOf prov:Entity},
establishing a uniform provenance spine. \texttt{mis:Thread} was the sole
exception, breaking this structural pattern and preventing PROV-O reasoners
from treating threads as provenance entities.

\textbf{Fix applied.}
\begin{lstlisting}
mis:Thread
    a owl:Class ;
    rdfs:subClassOf prov:Entity ;
    rdfs:label "Thread"^^xsd:string ;
    ...
\end{lstlisting}

% ── Bug 9 ────────────────────────────────────────────────────
\subsection{Bug~\#9 --- Placeholder Identity (\texttt{johndoe}) in Example Individuals}
\label{bug:9}

\textbf{Severity:} \badgeMinor\quad
\textbf{Category:} Publication Readiness\quad
\textbf{Status:} Fixed

\textbf{Description.}
The example user individual was declared as \ind{user\_johndoe} with
handle \texttt{johndoe} and display name \emph{John Doe} — a generic
placeholder. For a publication-ready ontology, examples should reference
a real, relevant person (the author is a natural choice) or use clearly
synthetic identifiers that cannot be confused with real accounts.
All five occurrences of the placeholder were found:

\begin{lstlisting}
misind:user_johndoe               # individual IRI
mis:userName  "johndoe"           # handle property
mis:userDisplayName "John Doe"    # display name
mis:hasAuthor misind:user_johndoe # in tweet_001
"RT @johndoe: ..."                # in tweet_002 text
\end{lstlisting}

\textbf{Fix applied.} All five occurrences replaced with Samriddha Pathak's identity:
\begin{lstlisting}
misind:user_samriddha_pathak
mis:userName        "samriddha_pathak"
mis:userDisplayName "Samriddha Pathak"
foaf:name           "Samriddha Pathak"
mis:userDescription "AI Researcher | Ontology Engineer | CAIR Nepal"
\end{lstlisting}

\subsection{Enhancement~\#13 --- Dedicated Label Individual for \texttt{tweet\_002}}
\label{enh:13}

\textbf{Severity:} \badgeEnhancement\quad
\textbf{Category:} OWL Design\quad
\textbf{Status:} Added

\textbf{Description.}
\texttt{mis:hasInformationLabel} is declared \texttt{owl:FunctionalProperty},
meaning each tweet may link to \emph{at most one} label instance.
Had \ind{tweet\_002} used the same label individual as \ind{tweet\_001},
both tweets would have been entailed to be the same individual
(\texttt{owl:sameAs}) by the FunctionalProperty semantics --- a critical
modelling error. Each tweet must have its own label individual.

\textbf{Fix applied.} A new individual \ind{label\_tweet002} was added:
\begin{lstlisting}
misind:label_tweet002
    a mis:InformationLabel ;
    mis:labelValue            "1"^^xsd:integer ;
    mis:annotationConfidence  "0.91"^^xsd:decimal ;
    mis:annotationTimestamp   "2024-06-10T14:00:00"^^xsd:dateTime ;
    mis:annotatedBy           misind:annotator_expert_01 ;
    mis:annotationMethod      misind:annotationEvent_001 .
\end{lstlisting}

\section{Recommendations for Future Versions}
% ============================================================

The following items are not errors in the current release but are
recommended for ontology version~1.1.0 or the camera-ready paper:

\begin{enumerate}[leftmargin=1.4em, itemsep=4pt]
  \item \textbf{Migrate base IRI from \texttt{http://} to \texttt{https://}.}
    The ontology IRI currently uses plain HTTP. Most Linked Data platforms
    now mandate HTTPS. A \texttt{owl:priorVersion} triple should be added
    to preserve backward compatibility.

  \item \textbf{Add SPARQL-based competency queries.}
    Include at least three representative SPARQL queries as
    \texttt{skos:example} or an external \texttt{.sparql} file to demonstrate
    ontology expressivity for reviewers.

  \item \textbf{Provide a Protégé \texttt{.owl} serialisation.}
    Some reviewers prefer OWL/XML or RDF/XML alongside Turtle.
    Both can be generated directly from Protégé with no content changes.

  \item \textbf{Register with the Linked Open Vocabularies (LOV) catalogue.}
    Submission to LOV increases discoverability and is expected by many
    Semantic Web conference programme committees.

  \item \textbf{Expand illustrative individuals.}
    The current sample covers Healthcare domain only. Adding one
    Political and one Educational domain tweet will better demonstrate
    the multi-domain claim of the ontology.
\end{enumerate}
